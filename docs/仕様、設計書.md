## 1. 概要

### システムの一言説明

```text
社内ドキュメントをGCSにアップロードするだけで
AIが根拠ドキュメント付きで日本語回答するQAシステム
```

### 解決する課題

```text
どの企業も抱えている課題
  ├── 「あの規定どこだっけ」が検索できない
  ├── 担当者に聞かないと分からない属人的な知識
  ├── ドキュメントが散在していて探せない
  └── ChatGPTに社内情報を入力するセキュリティリスク
```

### 提供する価値

```text
  ├── ドキュメントをアップロードするだけで即使える
  ├── 回答に根拠ドキュメントと該当箇所を明示
  ├── 社内データがGCP外に出ない（セキュア）
  └── 既存システムへの影響ゼロ
```

## 3. アーキテクチャ

### 全体フロー

```text
[ユーザー]
  └── PDFをGCSにアップロード
            ↓
[Cloud Run Job（Ingestion）]
  ├── GCSからドキュメント取得
  ├── テキスト抽出（PDF/Word/TXT）
  ├── チャンク分割（800文字・50文字オーバーラップ）
  ├── Vertex AI Embedding API でベクトル化
  ├── BigQuery documents テーブルに格納  ← Vector Search用
  └── Elasticsearch にインデックス登録   ← 全文検索用
            ↓
[Cloud Scheduler]
  └── 新規ドキュメント検知・日次自動実行
            ↓
[Cloud Run Service（QA API）]
  ├── 質問受信（POST /query）
  ├── Vertex AI Embedding でクエリベクトル化
  ├── ハイブリッド検索
  │    ├── BigQuery Vector Search（意味検索 Top-5）
  │    └── Elasticsearch 全文検索（キーワード検索 Top-5）
  ├── 検索結果をマージ・リランク
  ├── プロンプト組み立て
  └── Vertex AI Gemini で回答生成
            ↓
[レスポンス]
  └── 回答文 + 根拠ドキュメント + 該当箇所
```

### なぜElastic Searchを組み合わせるか

```text
Vector Search（意味検索）だけの問題
  └── 「有給」と入力しても「年次有給休暇」を
      見つけられないことがある
      └── 表記ゆれに弱い

全文検索だけの問題
  └── 「休みたい」と入力しても
      「有給休暇」を見つけられない
      └── 意味の近さに弱い

ハイブリッド検索
  └── 両方の弱点を補完
  └── 実務レベルの検索品質になる
  └── Elasticsearchの積極活用でスキル習得も兼ねる
```

## 4. データモデル

### BigQuery テーブル設計

```sql
-- dataset: doc_qa_dataset
-- table: documents

CREATE TABLE doc_qa_dataset.documents (
  id           STRING NOT NULL,         -- チャンクID（UUID）
  doc_id       STRING NOT NULL,         -- 元ドキュメントID
  doc_name     STRING NOT NULL,         -- ファイル名
  content      STRING NOT NULL,         -- チャンクテキスト
  chunk_index  INT64,                   -- チャンク番号
  page_number  INT64,                   -- ページ番号
  gcs_path     STRING,                  -- GCS上のパス
  embedding    ARRAY<FLOAT64> NOT NULL, -- Vertex AI Embedding（768次元）
  created_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Vector Index
CREATE VECTOR INDEX embedding_index
ON doc_qa_dataset.documents(embedding)
OPTIONS (distance_type = 'COSINE');
```

### Elasticsearch インデックス設計

```json
{
  "mappings": {
    "properties": {
      "id":          { "type": "keyword" },
      "doc_id":      { "type": "keyword" },
      "doc_name":    { "type": "keyword" },
      "content":     { "type": "text", "analyzer": "kuromoji" },
      "chunk_index": { "type": "integer" },
      "page_number": { "type": "integer" },
      "gcs_path":    { "type": "keyword" },
      "created_at":  { "type": "date" }
    }
  }
}
```

※ `kuromoji` アナライザーで日本語形態素解析

## 6. コンポーネント詳細設計

### Cloud Run Job（Ingestion）

```text
責務
  ├── GCSからドキュメントを取得
  ├── テキスト抽出
  │    ├── PDF  → pymupdf
  │    ├── Word → python-docx
  │    └── TXT  → そのまま読み込み
  ├── チャンク分割
  │    ├── 800文字単位
  │    └── 50文字オーバーラップ
  ├── Vertex AI Embedding API でベクトル化
  │    └── バッチ処理（100件/リクエスト）
  ├── BigQuery に格納
  └── Elasticsearch にインデックス登録

冪等性の設計
  └── doc_id + chunk_index で重複チェック
  └── 同じドキュメントを再投入しても二重登録されない
```

### Cloud Run Service（QA API）

```text
責務
  ├── クエリ受信
  ├── Vertex AI Embedding でクエリベクトル化
  ├── ハイブリッド検索
  │    ├── BigQuery Vector Search（意味検索）
  │    └── Elasticsearch 全文検索（kuromoji）
  ├── 結果マージ・リランク（RRF: Reciprocal Rank Fusion）
  ├── プロンプト組み立て
  └── Gemini で回答生成

プロンプト設計
  └── 「あなたは社内ドキュメントの専門家です。
        以下のドキュメントのみを根拠として
        日本語で正確に回答してください。
        根拠が見つからない場合は
        「該当する情報が見つかりませんでした」と答えてください。

        【参考ドキュメント】
        {検索結果Top-5}

        【質問】
        {クエリ}」
```

## 12. コスト見積もり（学習期間中）

```text
Vertex AI Embedding API
  └── ~$0.025 / 1000文字
  └── サンプルドキュメント程度なら数十円/回

Vertex AI Gemini
  └── 従量課金（呼び出し回数による）
  └── 学習用途なら数百円/月

BigQuery
  └── ストレージ: $0.02/GB/月 → ほぼ無料
  └── クエリ: 無料枠（10GB/月）内で収まる

Cloud Run
  └── 無料枠あり → ほぼ無料

Elastic Cloud
  └── 最小構成: ~$16/月（Starter）
  └── 使わないときは停止で節約

合計目安: 2,000〜3,000円/月
  └── Elastic Cloudが主なコスト
```
