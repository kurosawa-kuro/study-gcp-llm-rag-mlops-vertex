# 運用マニュアル

## 基本思想

このプロジェクトは **2コマンドで PDCA を高速で回す** 設計になっている。

```bash
make deploy-all     # 全環境構築（冪等・何度実行しても安全）
make destroy-all    # 全 GCP リソース削除（課金停止）
```

`deploy-all` を実行すれば、ツールインストールからデータ投入・検索可能状態まで全自動で完了する。
`destroy-all` を実行すれば、全 GCP リソースが削除され課金が停止する。
コードを変更したら `deploy-all` を再実行するだけでよい。

---

## 1. 初回のみ手動で行うこと

以下の2点だけは `deploy-all` の前に手動で準備が必要。一度配置すれば以降は触らない。

### 1-1. GCP 認証

```bash
gcloud init
```

### 1-2. クレデンシャル配置

```bash
vi env/secret/credentials.yml    # GCP SA キー + Elastic Cloud + Gemini APIキー
vi terraform/terraform.tfvars    # Terraform 変数
```

`terraform.tfvars` に必要な変数:

```hcl
elastic_cloud_api_key    = "<Elastic Cloud Org APIキー>"
google_ai_studio_api_key = "<Google AI Studio APIキー>"
```

ES接続用の username/password は `ec_deployment` から自動取得されるため手動設定不要。
Gemini APIキーは https://aistudio.google.com で取得。

---

## 2. deploy-all が実行すること

`deploy-all` → `bootstrap` の依存チェーンにより以下が順序実行される:

| Phase | ステップ | 内容 |
|-------|---------|------|
| 0 | `prereqs` | gcloud / terraform 未インストール時は自動インストール（インストール済みならスキップ） |
| 1 | `gcp-setup` | API 有効化 + SA 権限 + Vertex AI 権限 + Docker 認証 |
| 2 | `tf-bootstrap` | Terraform init → Registry import(冪等) → Registry 作成 |
| 3 | `images-push` | 全 Docker イメージ build + push（ingestion / api / eval） |
| 4 | `tf-apply` | 全 GCP リソース作成（Elastic Cloud 完了後に Cloud Run 作成 — `depends_on` で順序保証） |
| 5 | `upload-sample` | サンプルドキュメントを GCS にアップロード |
| 6 | `ingestion-run` | Ingestion Job 実行（チャンク分割 → Embedding → BQ + ES 格納） |
| 7 | `bq-vector-index` | BQ Vector Index 作成（5000行未満はスキップ・`Makefile` の `bq-vector-index` ターゲット） |

全ステップ冪等。何度実行しても安全。

## 3. 日常運用

### 3-1. ドキュメント追加

```bash
python3 scripts/ops/upload_doc.py <ファイルまたはディレクトリ>   # GCS にアップロード
make ingestion-run                                              # Ingestion Job 実行
```

Cloud Scheduler により毎日 9:00 JST に自動実行される。

### 3-2. QA クエリ

```bash
make qa-api-url                                                   # API URL 確認
python3 scripts/ops/query.py "有給休暇の申請手続きを教えてください"     # クエリ送信
python3 scripts/ops/query.py "出張の宿泊費上限は？" 3                  # top_k 指定
```

### 3-3. 監視・ログ

```bash
make qa-api-monitor                    # QA API /health チェック + Discord 通知
python3 scripts/monitor/ingestion.py   # Ingestion Job 結果確認 + Discord 通知
make qa-api-logs                       # QA API のログ
make ingestion-logs                    # Ingestion Job の実行履歴
```

Discord 通知を有効にするには `.env` に `DISCORD_WEBHOOK_URL` を設定。

## 4. 部分的な操作

`deploy-all` / `destroy-all` で足りない場合のみ使う。

### 4-1. アプリのみ再デプロイ

```bash
make deploy              # Ingestion + QA API 両方
make ingestion-deploy    # Ingestion のみ
make qa-api-deploy       # QA API のみ
```

### 4-2. テスト

```bash
make test              # 全テスト（Ingestion + QA API + shared + eval）
make ingestion-test    # Ingestion のみ
make qa-api-test       # QA API のみ
make shared-test       # shared/config テスト
make eval-test         # 評価指標・レポート テスト
```

### 4-3. Terraform

```bash
make tf-plan           # 差分確認
make tf-apply          # 全リソース反映
make tf-fmt            # フォーマット
make tf-validate       # 構文チェック
```

### 4-4. Docker ビルドのみ（push なし）

```bash
make ingestion-build
make qa-api-build
```

### 5-2. クレデンシャル

| ファイル | 内容 | git管理 |

## 6. RAG品質評価

> 有料プラン化済み。レートリミット到達時は自動リトライする。

### 6-1. 評価実行

```bash
make eval              # hybrid検索で評価実行（結果: scripts/eval/results/eval_YYYYMMDD.json）
make eval-baseline     # ベースライン記録（results/baseline_YYYYMMDD.json）
```

### 6-2. 検索パターン比較

```bash
make eval-search-patterns   # vector / elasticsearch / hybrid 3パターン比較
```

3パターンの評価を順次実行し、横並び比較レポートを出力する。

### 6-3. 評価指標

| 層 | 指標 | 説明 |
|---|------|------|
| 検索品質 | Recall@K (K=1,3,5,10) | 正解チャンクがTop-Kに含まれる割合 |
| 検索品質 | MRR | 最初の正解チャンクの順位の逆数の平均 |
| 回答品質 | Exact Match | 回答に期待キーワードが全て含まれる割合 |
| 回答品質 | ROUGE-L | 文字レベルLCSベースの回答網羅性 |

```bash

```


### 6-6. 評価パイプライン（Vertex AI Pipeline）

Vertex AI Pipelines 上で評価を自動実行するパイプライン。`src/pipeline/` に定義。

```bash
make eval-pipeline-run        # パイプライン実行（build + push 含む）
make eval-pipeline-schedule   # 週次スケジュール作成/更新（冪等・push 含む）
```

パイプラインは hybrid 検索で評価を実行し、結果を `gs://mlops-dev-a-doc-qa/eval-results` に保存する。

## 7. トラブルシューティング

基本方針: **困ったら `make destroy-all` → `make deploy-all` でやり直す。** 冪等なので壊れない。

それでも解決しない場合の調査手順:

### Ingestion Job が失敗する

```bash
make ingestion-logs    # 直近5件の実行履歴

# Cloud Logging で詳細ログ
gcloud logging read \
  'resource.type=cloud_run_job AND resource.labels.job_name=doc-qa-ingestion' \
  --limit=50 --format=json --project=mlops-dev-a
```

よくある原因:
- BQ ストリーミングバッファ → 数分待って再実行（自動スキップ対応済み）
- ES インデックス作成失敗 → アナライザー設定を確認

### QA API が応答しない

```bash
make qa-api-url        # URL 確認
make qa-api-monitor    # /health チェック
make qa-api-logs       # 直近ログ確認
```

よくある原因:
- GOOGLE_AI_STUDIO_API_KEY 未設定 → `terraform.tfvars` を確認して `make deploy-all`
- ES インデックス未作成 → `make deploy-all` で自動実行される

### Docker イメージが更新されない

```bash
docker build --no-cache -f src/api/Dockerfile -t TAG .
docker build --no-cache -f src/ingestion/Dockerfile -t TAG .
```

### Terraform apply が失敗する

```bash
make tf-validate       # 構文チェック
make tf-plan           # エラー詳細確認
make tf-fmt            # フォーマット修正
```
